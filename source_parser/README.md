# Source Parser: The Implementation Ground Truth

This package serves as the foundational "sensing" layer of the ingestion pipeline. While the project relies on the `clangd` index for high-level symbol relationships, `source_parser` provides the **Implementation Ground Truth** by directly analyzing source code using `libclang`.

## 1. Role and Rationale

Clangd's index is "declarative"—it knows *that* a symbol exists. However, it often lacks the deep structural details required for a high-fidelity Code Graph RAG. This package bridges that gap:

*   **Physical Boundaries**: Captures exact `body_location` coordinates (start/end lines) for implementation blocks, enabling precise source extraction for LLM summarization.
*   **Preprocessor Causality**: Tracks macro instantiations to link generated symbols (e.g., from X-Macros) back to their source `:MACRO` nodes via `[:EXPANDED_FROM]`.
*   **Implicit Hierarchy**: Proactively discovers "implicit" entities like template specializations that Clangd might omit, ensuring the graph hierarchy remains a fully connected tree.
*   **Include Graph**: Extracts the complete `(:FILE)-[:INCLUDES]->(:FILE)` graph, which is vital for transitive impact analysis during incremental updates.

## 2. High-Level Orchestration (manager.py)

The `CompilationManager` class is the single, unified interface for accessing source code metadata.

### Core Responsibilities
1.  **Orchestration**: Manages the transition from a list of files or a project folder to a validated set of `SourceSpan` and `IncludeRelation` objects.
2.  **Fast Path Discovery**: In Git repositories, it uses `git ls-tree` to instantly identify all project files at a specific commit, avoiding slow recursive filesystem walks.
3.  **Unified API**: Provides simple accessors (`get_source_spans()`, `get_include_relations()`, etc.) that abstract away the complexity of the underlying parsing workers.

## 3. The Parsing Engine (worker.py & node_parser.py)

The engine leverages `libclang` to parse source code with the exact same preprocessor context as the compiler.

### Advanced AST Traversal
*   **Stack-Based DFS**: Uses an explicit stack for AST traversal instead of recursion. This prevents stack overflow errors when encountering deeply nested C++ structures or massive macro expansions.
*   **Parent Anchoring**: When a member function is encountered, the parser semantically identifies its parent. If that parent (e.g., a class specialization) hasn't been processed yet, the parser proactively triggers its ingestion to maintain hierarchy integrity.
*   **Semantic Member Extraction**: The parser proactively identifies all immediate children of composite types (Classes, Enums, etc.) using the AST's semantic ownership rules. This correctly links members even when they are generated by macros in different files.

### Preprocessor Evidence
*   **Expansion Coordinates**: The engine uses coordinates where code was *expanded* rather than where characters were typed. This ensures that even macro-generated entities have unique, unambiguous identities.
*   **Coordinate Refinement**: Performs a targeted token scan to find the exact coordinates of a symbol's name within a macro-expanded signature, ensuring precise matching with the `clangd` index.

## 4. Identity and Resolution Strategies

The package uses **Identity Pluralism** to remain robust:

1.  **Semantic Identity (USR-Hash)**: Primarily uses USR-derived IDs (matching Clangd's internal logic: first 8 bytes of SHA1 hash of the USR). This allows O(1) matching between the parser and the index.
2.  **Syntactic Identity (Lexical)**: Falls back to a coordinate-based key (`kind::name::file_uri:line:col`) hashed via MD5 for nodes where a USR is unavailable (e.g., certain macro artifacts).

## 5. Caching and Performance

### Context-Aware Caching (`CacheManager`)
Parsing is expensive. `CacheManager` ensures we never do the same work twice:
*   **Git-Based**: Keyed by `new_commit` and `old_commit` hashes. Validated by checking the full hashes stored inside the cache file.
*   **Time-Based**: Fallback keyed by file modification times. Validated using a SHA256 hash of the entire file list to prevent false hits.
*   **Self-Healing**: Uses `safe_pickle_load` to automatically migrate legacy cache files to the new `source_parser` module structure if renames occur.

### Scalability
*   **"Spawn" Strategy**: Uses `ProcessPoolExecutor` with the `spawn` method. This starts fresh Python interpreters for workers, drastically reducing memory overhead and preventing heap fragmentation in large projects.
*   **Deterministic TU Hashing**: Buckets compiler flags into categories (Macros, Includes, etc.) to create a stable MD5 hash of the preprocessor environment. If two Translation Units share the same environment, they can share the same header cache.
*   **Header Deduplication**: Maintains a worker-level registry of processed headers to avoid re-parsing standard headers (like `iostream`) thousands of times.
