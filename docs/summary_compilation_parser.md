# Algorithm Summary: `compilation_parser.py`

## 1. Role in the Pipeline: The Implementation Ground Truth

This module serves as the foundational "sensing" layer of the ingestion pipeline. While the pipeline relies heavily on the `clangd` index for high-level symbol relationships, Clangd is primarily designed for IDE features like navigation and completion. It often simplifies or omits the deep structural implementation details required for a high-fidelity Code Graph RAG. The `CompilationParser` exists to bridge this gap by providing the **Implementation Ground Truth**.

### 1.1. Bridging the Semantic-Lexical Gap
Clangd's index is "declarative"—it knows *that* a function exists and *where* its name is declared or defined. However, it frequently lacks the precise physical boundaries (the "body spans") of that function's implementation. This is especially true in C++, where implementations can be hidden inside complex template specializations or macro expansions. 
*   **Retaining body_location**: This module performs a full AST (Abstract Syntax Tree) traversal to capture the exact start and end line/column coordinates of every implementation block. 
*   **Enabling RAG**: This is critical for the downstream RAG (Retrieval-Augmented Generation) stages. Without these precise spans, an AI agent would only know the name of a function but would have no reliable way to extract its full source code for summarization or analysis.

### 1.2. Capturing Preprocessor Causality
In codebases like LLVM, the preprocessor is used as a meta-programming engine. A single macro call can generate entire hierarchies of classes and methods. 
*   **The Macro Evidence Problem**: Clangd typically indexes the resulting symbols but loses the connection to the source macro. An AI agent seeing a generated class name like `WaitEventType` might be confused because that string does not appear literally in the source file.
*   **Causality Metadata**: This module tracks `MACRO_INSTANTIATION` cursors during the AST walk. It links generated symbols to their source macros via `expanded_from_id` and captures the `original_name` (the raw invocation text, e.g., `AMDGPU_DECLARE_WAIT_EVENTS(WaitEventType)`). This allows the graph to "explain" the existence of magic symbols.

### 1.3. Proactive Hierarchy Discovery
Large-scale C++ often involves "implicit" entities. For instance, a member function specialization like `template <> void Box<int>::open()` implies the existence of a class specialization `Box<int>`, even if that class is never explicitly declared in the code.
*   **Parent Anchoring**: Our parser implements "Parent Anchoring" logic. When it encounters a member, it semantically identifies its parent. If that parent is a specialization that Clangd missed, the parser proactively processes it, ensuring the graph hierarchy remains a fully connected tree rather than a collection of orphaned implementation snippets.

### 1.4. Normalization and Interface Abstraction
The module acts as a "Strategy" layer. It encapsulates the significant complexity of the `libclang` and `tree-sitter` APIs. 
*   **Unified SourceSpans**: It normalizes disparate AST representations into a unified `SourceSpan` structure. 
*   **Decoupling**: This allows high-level logic (like graph building and caching) to remain agnostic to the specific parser being used, ensuring the system can be extended to new languages or parsing engines without a total rewrite.

## 2. Core Design: Semantic and Syntactic Identities

The module is architected around the concept of **Identity Pluralism**. It recognizes that different parsing engines have different "perceptions" of a symbol's identity. To remain robust and modular, the system supports two fundamentally different identity strategies.

### 2.1. Semantic Identity (The USR-Hash Strategy)
For high-fidelity semantic parsers like `ClangParser`, the system uses **USR-derived IDs** as the primary identity for all AST nodes.

*   **The Rationale**: A USR (Unified Symbol Resolution) is a unique string generated by the compiler that encodes the semantic signature of an entity (e.g., its namespace, class name, function name, and parameter types). Unlike source coordinates (line and column), a USR is stable across code edits—adding a comment at the top of a file changes all line numbers but does not change the USR of the functions below.
*   **The Hash Algorithm**: To ensure 100% compatibility with the Clangd index, we replicate Clangd's internal ID generation logic. We take the **SHA1 hash** of the USR string and truncate it to the **first 8 bytes**, represented as a 16-character uppercase hex string (e.g., `17E079A8B4D313EB`). 
*   **The Benefit**: By generating these IDs in the parser, matching becomes an **O(1) dictionary lookup**. If the parser and Clangd both see the same semantic entity, they will generate the exact same ID, eliminating the need for complex coordinate-based heuristics.

### 2.2. Syntactic Identity (The Lexical Strategy)
For syntactic parsers like `TreesitterParser`, or for Clang nodes where a USR is unavailable (such as specific macro-generated artifacts), the system falls back to **Lexical Identity**.

*   **The Rationale**: Syntactic parsers analyze code as a tree of tokens without performing full semantic resolution or header inclusion. They lack the context to generate USRs.
*   **The Mechanism**: The module uses a coordinate-based `node_key` format: `kind::name::file_uri:line:col`. This key is then MD5-hashed into a 32-character "synthetic ID".
*   **Authority of Expansion Locations**: Crucially, these keys use **Expansion Coordinates** (where the code was expanded) rather than **Spelling Coordinates** (where the characters were typed). In a single Translation Unit (TU), it is physically impossible for two distinct implementation entities of the same kind and name to occupy the same expansion point. This ensures that even our coordinate-based identities are unambiguous.

### 2.3. Strategy Pattern Integration
The `CompilationParser` base class (ABC) enforces this dual-identity approach through a standardized interface.
*   **`make_symbol_key` and `make_synthetic_id`**: Provided as utility methods to ensure all coordinate-based fallback identities are generated consistently across all parser implementations.
*   **`parser_kind_to_index_kind`**: A normalization method that maps parser-specific vocabulary (e.g., libclang's `FUNCTION_TEMPLATE` or Tree-sitter's `function_definition`) into the standard "Index Vocabulary" (e.g., "Function", "StaticMethod"). This ensures that regardless of the parser used, the resulting IDs and Kind strings are ready for the enrichment phase.

## 3. Concrete Strategy: `ClangParser`

This is the primary parsing strategy, valued for its semantic accuracy. It leverages `libclang` to parse source code with the exact same preprocessor context as the compiler. However, `libclang`'s AST representation has several "gaps" that the `ClangParser` must bridge using advanced heuristics.

### 3.1. Technical Nuances in C++ Mapping

#### 3.1.1. Context-Aware Method Identification
In the Clang AST, template member functions are often generalized under the `FUNCTION_TEMPLATE` kind. This is problematic because Clangd's index differentiates between a top-level `Function` and a `StaticMethod` or `InstanceMethod` inside a class.
*   **The Logic**: In `_convert_node_kind_to_index_kind`, the parser performs a semantic context check. If a function or template function has a `semantic_parent` that is a composite type (Class, Struct, Union, or ClassTemplate), the parser re-classifies it. It then uses `node.is_static_method()` to determine if it should be a `StaticMethod` or `InstanceMethod`.
*   **Rationale**: Without this context-awareness, synthetic IDs for methods would be generated with the wrong "Kind" string, causing them to fail direct matching with the Clangd index.

#### 3.1.2. Template Keyword Resolution
A significant limitation of `libclang` is that it identifies all templated types (classes, structs, and unions) as `CLASS_TEMPLATE`. This loses the syntactic distinction required to match the index correctly.
*   **The Heuristic**: The parser implements a mini-state-machine using token scanning. It tokenizes the node's implementation header and tracks "bracket depth" to skip the template parameter list (`template <...>`). Immediately after the closing `>`, it scans the next few tokens to find the actual defining keyword (`struct`, `class`, or `union`).
*   **Efficiency**: To avoid the massive overhead of tokenizing entire classes, this scan is performed using `islice(node.get_tokens(), 100)`, limiting the analysis to the first 100 tokens where the keyword is guaranteed to exist.

#### 3.1.3. Parent Anchoring: Proactive Hierarchy Building
Clangd often omits index entries for "implicit" template specializations. For example, if a user defines a member specialization like `template <> void Box<int>::display()`, the index will contain the method `display`, but might not contain the parent class `Box<int>`. This leads to "orphaned" methods in the graph.
*   **The "Pull-Up" Recursion**: The `_get_parent_id` method implements **Parent Anchoring**. When it identifies a semantic parent that is a composite type but has not yet been processed in the current TU context, it **proactively triggers** `_process_generic_node` for that parent.
*   **The Result**: This ensures that every member function "pulls" its parent into the `SourceSpan` collection. Even if the parent doesn't exist in the index, it will be added as a **Synthetic Symbol**, ensuring the graph hierarchy remains a fully connected tree.

#### 3.1.4. Precise Coordinate Resolution (`_get_symbol_name_location`)
By default, `libclang` may report the expansion location of a macro-expanded symbol as the start of the macro call. This coordinate is too broad for precise matching with Clangd.
*   **The Fix**: The parser tokenizes the node and performs a targeted search for the token whose spelling matches the symbol's name (`node.spelling`). It then retrieves the expansion location of that **specific token**. This ensures that even for complex macro-generated signatures, the `name_location` points precisely to the implemention characters.

#### 3.1.5. Anonymity Management and the ID Bridge
Anonymous enums, structs, and unions present a naming challenge.
*   **The Strategy**: The parser detects anonymity via spelling patterns (e.g., `(unnamed enum at...)`). For these nodes, it uses the **raw USR string** as the node's name for debug visibility.
*   **The Bridge**: Because USRs are stable, the USR-hash ID of the anonymous parent matches the `Type` field of its children (like `EnumConstant`) in the Clangd index. This allows the `SourceSpanProvider` to bridge members back to their anonymous parents mathematically, even when they are defined in separate files (via `#include` in an anonymous enum).


### 3.2. Performance and Scalability Architecture

Parsing thousands of Translation Units (TUs) in a large project like LLVM requires significant architectural discipline to manage memory and CPU time effectively.

#### 3.2.1. The "Spawn" Worker Strategy
The `ClangParser` uses `concurrent.futures.ProcessPoolExecutor` with the `spawn` start method (via `multiprocessing.get_context("spawn")`).
*   **Rationale**: In a large build, the main Python process's memory footprint can exceed several gigabytes. The default `fork` method copies the entire memory space of the parent into every child worker. In contrast, `spawn` starts fresh Python interpreters. 
*   **Impact**: This drastically reduces the cumulative memory usage of the worker pool and prevents heap fragmentation, ensuring the pipeline can scale to 10,000+ TUs without crashing.

#### 3.2.2. Deterministic TU Hashing (`_get_tu_hash`)
A critical part of the caching layer is the ability to uniquely identify a compilation environment.
*   **The Problem**: Compiler flags can be passed in any order (`-std=c++17 -DDEBUG` vs `-DDEBUG -std=c++17`), and many flags (warnings, optimizations) do not affect the AST structure.
*   **The Logic**: The parser buckets flags into five categories: Language Standards, Macros, Features, Includes, and Other. It preserves the relative order *within* buckets (essential for include search paths) but normalizes the order *between* buckets.
*   **Result**: This creates a deterministic MD5 hash of the preprocessor environment. If two TUs share the same environment, they can share the same header implementation cache.

#### 3.2.3. Header Caching and Deduplication
To avoid parsing the same header file thousands of times, the module implements a two-tier cache:
*   **`_global_header_cache`**: A shared registry (per worker process) of headers already fully processed for a specific `tu_hash`.
*   **`_should_process_node`**: Before traversing a header's AST, the worker checks this registry. If the header was already parsed in an identical preprocessor context, the traversal is skipped.
*   **Volatile Headers**: Files ending in `.inc`, `.def`, or `.gen` (often used for X-Macros) are excluded from this cache. This ensures that context-specific symbols generated by multiple inclusions are always captured accurately.

#### 3.2.4. Stack-Based DFS (`_walk_ast`)
Standard recursive AST traversal is prone to stack overflow in C++, especially when encountering massive `switch` statements or deeply nested macro expansions (e.g., `SemaARM.cpp` contains chains exceeding 5,000 nodes).
*   **The Logic**: The `_walk_ast` method uses an explicit Python `list` as a stack. It iteratively pops a node, processes it via `_process_node`, and then pushes its children onto the stack in reverse order.
*   **Result**: This iterative approach maintains identical Depth-First Search (DFS) visit order to the recursive version but is immune to Python's recursion depth limits.

#### 3.2.5. Memory-Efficient Data Structures
The parser returns implementation data in a nested format: `Dict[file_uri, Dict[id, SourceSpan]]`.
*   **Rationale**: By using 16-character hex IDs as keys instead of long coordinate-based strings, the memory overhead of the in-memory span dictionary is significantly reduced. This is vital when merging results from thousands of files into the final `CompilationManager` result.

## 4. Macro Causality and Member Extraction Logic

This section details the logic used to explain "magic symbols" generated by the preprocessor and how the parser captures semantic ownership that transcends physical file boundaries.

### 4.1. Macro Causality Tracking (`_get_macro_causality`)
When a symbol is generated by a macro (e.g., `DECLARE_MODULE(FS)`), it physically resides at the expansion site, but its semantic origin is the macro definition. The parser reconstructs this link using geometric analysis.

#### 4.1.1. Recording Instantiations
As the `ClangParser` walks the AST, it maintains a registry of every `MACRO_INSTANTIATION` node found in the Translation Unit. For each instantiation, it records the `extent` (the exact start and end coordinates in the source file).

#### 4.1.2. The `extent_contains` Geometric Check
When a generic node (Function, Class, etc.) is identified as a definition, the parser invokes `_get_macro_causality`. This method performs a coordinate comparison:
*   **The Logic**: It iterates through all recorded macro instantiations in the same file. It checks if the symbol's physical `extent` is fully contained within the macro instantiation's `extent`.
*   **The Outermost Rule**: A symbol can be nested within multiple macros (e.g., `MACRO_A` calling `MACRO_B`). The parser identifies the **outermost** macro that fully encloses the symbol. This aligns the causality with the user's top-level intent in the source code.

#### 4.1.3. Evidence Extraction
Once the enclosing macro is found, the parser extracts two pieces of "evidence":
1.  **`expanded_from_id`**: The USR-hash ID of the macro definition itself. This allows the graph to draw an `[:EXPANDED_FROM]` relationship.
2.  **`original_name`**: The raw source text of the macro call (e.g., `AMDGPU_DECLARE_WAIT_EVENTS(WaitEventType)`). This is extracted by reading the implementation file at the macro instantiation coordinates.

### 4.2. Semantic Member Extraction (`member_ids`)
Traditional lexical containment fails when a composite type (like an Enum or Class) uses macros to define its members, especially when those macros are defined or expanded in different files (the "X-Macro" pattern).

#### 4.2.1. The Child Registry
In `_process_generic_node`, if the node is a composite type (`NODE_KIND_FOR_COMPOSITE_TYPES`), the parser proactively iterates through its **immediate semantic children** via `node.get_children()`. 

#### 4.2.2. Filtering and ID Collection
Not all children are relevant members. The parser filters the children against the `NODE_KIND_MEMBERS` constant, which includes:
*   Fields and Variables (static members)
*   Methods and Functions
*   Enum Constants
*   Nested Types (Classes, Structs, Type Aliases)

For every valid member found, the parser computes its **USR-derived ID** and stores it in the parent's `member_ids` list.

#### 4.2.3. Why this is Robust
This logic captures the **semantic ownership** defined by the compiler's AST. Even if a macro expands to an `EnumConstant` whose coordinates are in a different file or at a distant line number, the `ClangParser` sees that it is semantically owned by the parent `Enum`. This list is then used by the `SourceSpanProvider` as the "ultimate truth" for linking members to their parents, bypassing the limitations of coordinate-based containment.
