# Orchestration & Infrastructure

The `summary_engine` coordinates massive parallelism with thread-safe state management through a modular Map-Reduce architecture.

## 1. The Map-Reduce Engine (`_parallel_process`)
The orchestrator implements a custom Map-Reduce pattern to process batches of nodes efficiently while maintaining data integrity.

1.  **Map Phase (Parallel)**: Worker functions are dispatched to a `ThreadPoolExecutor`. This phase is dedicated to I/O-bound LLM API calls and source code retrieval.
2.  **Reduce Phase (Serial)**: As workers complete, their results are processed **serially in the main thread**.
    *   **Data Validation**: Results are inspected to ensure only valid, non-empty summaries populate the cache.
    *   **Atomic Cache Mutation**: The `SummaryCacheManager` is updated in a single-threaded loop, eliminating the need for complex locks and preventing race conditions.
    *   **Telemetric Reduction**: Performance statistics (restored, generated, unchanged) are updated in real-time.

## 2. Runtime Status Tracking & Dependency Propagation
The `SummaryCacheManager` maintains a transient, in-memory **`runtime_status`** dictionary that is reset at the start of every run. This mechanism ensures that changes in leaf nodes correctly propagate up the project hierarchy.

### 2.1 The Status Flags
*   **`visited`**: Set to `True` whenever a node is processed. In "Builder" mode, the engine uses this to prune entries for symbols or files that have been deleted from the project.
*   **`code_analysis_changed`**: Set only when a function's physical implementation analysis is regenerated by the LLM.
*   **`summary_changed`**: Set when any node's final summary is regenerated. This flag triggers the "Waterfall" staleness check in all parent or dependent nodes.

## 3. The Waterfall Logic
The `NodeSummarizer` follows a strict "Principle of Least Effort" to minimize API costs and latency:

1.  **DB Check**: If `current_hash == db_code_hash` AND context is clean (`summary_changed` flags are False for all dependencies), the node is skipped (`unchanged`).
2.  **Cache Check**: If the database is stale, the engine checks the L1 Cache. If the cached hash matches the current physical hash and context is clean, the summary is restored to the DB without an LLM call.
3.  **Regeneration**: If all prior stages fail, an LLM call is triggered.

## 4. Dynamic Model-Aware Resource Adaptation
The engine automatically optimizes its chunking and summarization strategy based on the specific LLM model being used.

*   **Implementation**: Uses `litellm.get_model_info(model)` to retrieve the `max_input_tokens` limit.
*   **Discovery Priority**:
    1.  User-specified `args.max_context_size`.
    2.  Model-reported `max_input_tokens` (Granular limit).
    3.  Model-reported `max_tokens` (Legacy total limit).
    4.  Default to `128,000` tokens.
*   **Optimization**: Large context windows (128k+) allow the engine to summarize complex classes or files in a single pass, while smaller models automatically trigger iterative chunking.

## 5. Interface Analysis (Pass 1 Integrated)
To achieve 100% coverage, functions without bodies (declarations/virtuals) are summarized deterministically without LLM costs.
*   **Mechanism**: The engine generates a concise contract description based on the Name, Signature, and Return Type.
*   **Identity Hashing**: The `node_id` is used as the `code_hash`. If a function gains or loses a body, the hash mismatch triggers an automatic switch between deterministic generation and LLM-powered implementation analysis.

## 6. Cache Integrity & "Promote-on-Success"                                                              
The L1 cache (`summary_backup.json`) is protected by a safety-first persistence strategy: 
1.  **Intermediate Saves**: Writes to `.tmp` files after every pass.     
2.  **Sanity Check**: Verifies that the new cache size hasn't dropped significantly (preventing data loss due to empty DBs or network failures).                
3.  **Rolling Backups**: Maintains two levels of history (`.bak.1`, `.bak.2`).
