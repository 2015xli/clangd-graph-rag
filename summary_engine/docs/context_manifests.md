# Structural Manifests & Physical Hashing

To ensure 100% summary coverage and accurate semantic context, high-level nodes (Classes, Data Structures, Files, Namespaces) are summarized using a "Manifest" approach rather than simple child roll-ups.

## 1. Structural Manifests (Pass 3)
The engine treats `CLASS_STRUCTURE` (C++ classes/structs) and `DATA_STRUCTURE` (C structs, enums, unions) as unified structural entities.

### 1.1 Content Integration
Summaries are generated by combining three sources of truth:
*   **Physical Code**: The literal source code from the node's `body_location`.
*   **Macro Origins**: If a symbol is expanded from a macro, the summary identifies the macro origin (`original_name`), providing "evidence" for the symbol's existence.
*   **Member Inventory**: A list of all methods and fields (including out-of-line members physically located in other files) and their individual summaries.
*   **Template Context**: Explicitly distinguishes between primary templates (`template_params`) and specializations (`specialization_args`).

### 1.2 Synthetic Classes
Phony classes created for member specializations (which share the blueprint's extent) are explicitly identified as "specialized structural containers." Their summaries explain how the template behaves in that specific type context.

## 2. Namespace Manifests (Pass 4)
Namespaces are distributed logical containers. Their summaries are based on a **Membership Manifest**.
*   **Content**: A list of all logical children (`SCOPE_CONTAINS` and `DEFINES_TYPE_ALIAS`).
*   **Type Alias Enrichment**: The manifest includes the **`aliased_canonical_spelling`** for type aliases (e.g., `status_t` alias of `int`), helping the LLM understand the underlying data types.

## 3. File & Folder Manifests (Pass 5 & 6)
Files and folders provide a "Structural Inventory" of their contents.
*   **File Manifest**: Includes relative paths of header files (`[:INCLUDES]`), literal source code context, and a categorized inventory of symbols.
*   **Folder Manifest**: Describes the organizational purpose of subfolders and files.
*   **Empty Nodes**: If no source components are detected, a deterministic string is returned (e.g., "This folder is empty or contains no recognized source code"), ensuring no node has a null summary.

## 4. Physical Staleness Tracking (`code_hash`)
The engine uses an MD5 `code_hash` to ensure summaries are only regenerated when the underlying physical state changes.

### 4.1 On-the-fly Hashing
Classes and Files calculate their `code_hash` **during their respective summarization passes** by reading their literal source code. This avoids redundant filesystem reads while ensuring that changes to definitions or file-level macros correctly invalidate the summary.

### 4.2 Membership Hashing (Namespaces)
Since namespaces lack a physical body, they use a **Membership Hash** based on their children's identities.
*   **Identity String**: `"{id}:{name}:{aliased_canonical_spelling}"`.
*   **Calculation**: The engine sorts these strings for all children and computes a digest. Any change to the namespace membership (adding/removing/renaming members) triggers a staleness check.

### 4.3 Database Synchronization
All newly calculated hashes are persisted to the Neo4j database upon successful summarization, enabling O(1) physical staleness checks in subsequent runs.
