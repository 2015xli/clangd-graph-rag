# Summary Engine: AI-Powered Code Summarization

This package contains the core intelligence and data persistence layers for the RAG (Retrieval-Augmented Generation) generation process. It is responsible for analyzing source code via LLMs, generating hierarchical summaries, and managing a robust caching system to ensure performance and cost-efficiency.

## 1. Orchestration Layer (orchestrator.py)

The `SummaryEngine` acts as the central logistics coordinator. It determines the *order* and *scope* of work but delegates the complex *logic* for processing each node to the `NodeSummaryProcessor`.

### Core Architectural Principles
*   **Separation of Concerns**: Orchestrator handles DB I/O and parallelism; Processor handles LLM logic and staleness checks.
*   **Hierarchical Order**: Enforces bottom-up or level-by-level processing for nodes with dependencies (e.g., classes must be summarized after their parent classes).
*   **Map-Reduce Engine**: Uses `_parallel_process` to dispatch LLM calls to a thread pool (Map) and then serially validates and commits results to the cache and database (Reduce).
*   **Thread-Safe State**: All modifications to the `SummaryCacheManager` happen serially in the main thread to prevent race conditions.

## 2. Logic Layer: Node Processing (node_summarizer.py)

The `NodeSummaryProcessor` is the stateless "brain" of the system. It implements a **"Waterfall" Decision Process** for every node:

1.  **DB Hit**: Is there a valid, non-stale summary already in Neo4j? (Perfect state, no work).
2.  **Cache Hit**: Is there a valid, non-stale summary in the L1 cache? (Restore to DB, no LLM call).
3.  **Regeneration**: If stale or missing, construct a prompt and call the LLM API.

### Handling Large Contexts
If a node's context (e.g., a massive class or a long function) exceeds the LLM's token limit, the processor uses **Iterative Summarization**:
*   It chunks the text/dependencies into overlapping segments.
*   It processes them sequentially, feeding the running summary from the previous chunk into the next prompt.

## 3. Data Layer: Cache Management (node_cache.py)

The `SummaryCacheManager` is the single source of truth for cached data (`code_analysis`, `summary`, and `code_hash`).

### "Promote-on-Success" Strategy
To ensure a good cache is never corrupted by a failed run:
1.  **Intermediate Saves**: Writes to `.tmp` files after each major pass.
2.  **Sanity Check**: Before finalizing, it compares the new cache size against the old one. If the new cache is suspiciously smaller (e.g., < 95% of previous size), it **aborts the promotion** and logs a critical error.
3.  **Rolling Backups**: Maintains a 2-level backup rotation (`.bak.1`, `.bak.2`).

### Automatic Maintenance
*   **Pruning**: In "builder" mode, it removes entries for nodes that no longer exist in the project.
*   **Self-Healing**: Automatically migrates legacy pickle files if module names change.
*   **Fake Detection**: Detects and purges summaries generated by the "fake" LLM client when a real LLM API is activated.

---

## 4. Logical Mixins

The engine's logic is modularized into mixins based on node categories:
*   **`FunctionProcessorMixin`**: Handles individual code analysis (Pass 1) and contextual caller/callee summaries (Pass 2).
*   **`ScopeProcessorMixin`**: Manages logical hierarchies like C++ Namespaces and Class Inheritance levels.
*   **`HierarchyProcessorMixin`**: Handles physical roll-ups: File -> Folder -> Project.

---

## CLI Usage
Manage the L1 cache directly:
```bash
# Backup summaries from Neo4j to cache file
python3 -m summary_engine backup

# Restore summaries from cache file to Neo4j
python3 -m summary_engine restore

# Surgically remove all "fake" summaries from DB and cache
python3 -m summary_engine clean-fakes
```
